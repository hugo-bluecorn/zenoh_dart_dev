# Phase 16: Bytes Serialization/Deserialization — Revised Design Spec

**Status:** DRAFT
**Author:** CA (Riker)
**Date:** 2026-03-30
**Source of truth:** `development/phases/phase-16-bytes.md`
**Predecessor:** Phase 14 (Throughput Benchmarks, PR #28)

---

## Goal

Implement the full bytes serialization/deserialization API. This enables
cross-language typed data exchange — a C publisher serializing a uint32 can
be deserialized correctly by a Dart subscriber, and vice versa.

Three API layers matching zenoh-c:
1. **Convenience** — `ZBytes.fromInt()` / `toInt()` for single-value round-trips
2. **Composite** — `ZSerializer` / `ZDeserializer` for multi-value payloads (sequences, key-value pairs)
3. **Raw** — `ZBytesWriter` / slice iterator for byte-level assembly and fragmented reading

No network operations — pure serialization, validated by the `z_bytes.dart`
CLI example mirroring `extern/zenoh-c/examples/z_bytes.c`.

**Note on C shim naming:** `zd_serializer_serialize_string` wraps
`ze_serializer_serialize_str` (null-terminated C string), not
`ze_serializer_serialize_string` (which takes `z_loaned_string_t*`).
The Dart-facing name follows our convention; the zenoh-c function being
wrapped is `_str`. CP and CI should be aware of this mapping.

## Why C Shim Wrappers Are Required

The `ze_serialize_*`, `ze_deserialize_*`, `ze_serializer_*`, and
`ze_deserializer_*` functions are exported from libzenohc.so (not macros).
However, our architecture routes all FFI calls through libzenoh_dart.so
with `zd_*` symbols generated by ffigen. The `ze_*` symbols are not in
our bindings. Additionally:

- **Deserialize functions** take `z_loaned_bytes_t*` which requires the
  `z_loan()` macro — FFI barrier pattern #2
- **Serializer finish** takes `ze_moved_serializer_t*` — FFI barrier pattern #1
- **Writer finish** takes `z_moved_bytes_writer_t*` — FFI barrier pattern #1
- **Deserializer creation** returns `ze_deserializer_t` by value — our pattern
  allocates on heap and fills via pointer
- **Opaque type sizes** for `ze_owned_serializer_t`, `ze_deserializer_t`,
  `z_owned_bytes_writer_t`, `z_bytes_slice_iterator_t` — FFI barrier pattern #4

## Design Decision: No One-Shot C Shim Functions

The original spec proposes 26 one-shot C shim functions (`zd_serialize_<type>`
/ `zd_deserialize_<type>`). These are unnecessary. The convenience methods
on `ZBytes` (`fromInt`, `toInt`, etc.) use `ZSerializer` / `ZDeserializer`
internally in Dart:

```dart
static ZBytes fromInt(int value) {
  final ser = ZSerializer();
  ser.serializeInt64(value);
  return ser.finish();
}

int toInt() {
  final deser = ZDeserializer(this);
  final value = deser.deserializeInt64();
  if (!deser.isDone) {
    throw ZenohException('Payload contains more bytes than expected');
  }
  deser.dispose();
  return value;
}
```

This mirrors the C++ approach where `zenoh::ext::serialize(value)` creates
a `Serializer` internally. Saves 26 C shim functions.

## Design Decision: Reader Deferred

The `z_bytes_reader_*` API (seek, tell, read, remaining) provides random
access into byte payloads. This is an advanced feature — `ZBytes.toBytes()`
already reads the full payload as `Uint8List`, and the slice iterator handles
fragmented data. The reader can be added in a future patch if needed.

### Slice Iterator (3)

| Function | zenoh-c API | Purpose |
|----------|-------------|---------|
| `zd_bytes_slice_iterator_sizeof` | `sizeof(z_bytes_slice_iterator_t)` | Opaque type size |
| `zd_bytes_get_slice_iterator` | `z_bytes_get_slice_iterator(z_loan(...))` | Create iterator from loaned bytes |

**Slice iterator view semantics:** `z_bytes_slice_iterator_next` outputs
a `z_view_slice_t` — a borrowed view, NOT an owned copy. The view is valid
only until the next `iterator_next` call or until the source `ZBytes` is
disposed. The C shim must extract `z_slice_data()` and `z_slice_len()` from
the view and return them to Dart (pointer + length). The Dart implementation
must copy the data into a new `Uint8List` per-step before returning it to
the caller. Do NOT retain the pointer across iterations.

## Dart API Surface

### New: `ZSerializer` (`package/lib/src/serializer.dart`)

Streaming serializer for composing multi-value payloads. Wraps
`ze_owned_serializer_t`.

```dart
class ZSerializer {
  ZSerializer();

  // Arithmetic types — all widths for cross-language interop
  void serializeUint8(int value);
  void serializeUint16(int value);
  void serializeUint32(int value);
  void serializeUint64(int value);
  void serializeInt8(int value);
  void serializeInt16(int value);
  void serializeInt32(int value);
  void serializeInt64(int value);
  void serializeFloat(double value);
  void serializeDouble(double value);
  void serializeBool(bool value);

  // Compound types
  void serializeString(String value);
  void serializeBytes(Uint8List value);

  // Sequence support — serialize length header, then serialize each element
  void serializeSequenceLength(int length);

  /// Finalize serialization and return the payload.
  /// Consumes the serializer — cannot be used after calling finish().
  ZBytes finish();

  /// Release without finishing (error cleanup).
  void dispose();
}
```

### New: `ZDeserializer` (`package/lib/src/deserializer.dart`)

Streaming deserializer for reading multi-value payloads. Wraps
`ze_deserializer_t` (copyable stack struct, heap-allocated by C shim).

```dart
class ZDeserializer {
  /// Create a deserializer for the given bytes.
  /// The deserializer borrows from the loaned bytes — disposing the ZBytes
  /// while the deserializer is in use is undefined behavior. The constructor
  /// stores a reference to the ZBytes and checks _ensureNotDisposed() on
  /// each deserialization call to prevent use-after-free.
  ZDeserializer(ZBytes bytes);

  // Arithmetic types
  int deserializeUint8();
  int deserializeUint16();
  int deserializeUint32();
  int deserializeUint64();
  int deserializeInt8();
  int deserializeInt16();
  int deserializeInt32();
  int deserializeInt64();
  double deserializeFloat();
  double deserializeDouble();
  bool deserializeBool();

  // Compound types
  String deserializeString();
  Uint8List deserializeSlice();

  // Sequence support
  int deserializeSequenceLength();

  /// Check if all data has been consumed.
  bool get isDone;

  /// Free the heap-allocated deserializer struct.
  void dispose();
}
```

### New: `ZBytesWriter` (`package/lib/src/bytes_writer.dart`)

Raw byte assembler for building multi-part payloads. Wraps
`z_owned_bytes_writer_t`.

```dart
class ZBytesWriter {
  ZBytesWriter();

  /// Write raw bytes.
  void writeAll(Uint8List data);

  /// Append an existing ZBytes (consumed — the ZBytes cannot be used after).
  void append(ZBytes bytes);

  /// Finalize and return the assembled payload.
  /// Consumes the writer — cannot be used after calling finish().
  ZBytes finish();

  void dispose();
}
```

### Modified: `ZBytes` (`package/lib/src/bytes.dart`)

Add convenience methods for single-value round-trips and slice iteration.

```dart
class ZBytes {
  // Existing: fromString, fromUint8List, toStr, toBytes, clone, isShmBacked

  // New: single-value convenience (uses ZSerializer/ZDeserializer internally)
  static ZBytes fromInt(int value);     // serializes as int64
  static ZBytes fromDouble(double value); // serializes as double
  static ZBytes fromBool(bool value);   // serializes as bool

  int toInt();       // deserializes as int64, checks isDone
  double toDouble(); // deserializes as double, checks isDone
  bool toBool();     // deserializes as bool, checks isDone

  // New: slice iteration for fragmented data
  Iterable<Uint8List> get slices;
}
```

**`toInt()` / `fromInt()` mapping:** Uses int64 (Dart's native integer width).
For other widths, use `ZSerializer` / `ZDeserializer` explicitly.

**uint64 overflow caveat:** Dart's `int` is a signed 64-bit integer (range
-2^63 to 2^63-1). `serializeUint64` / `deserializeUint64` cannot represent
values in the range 2^63 to 2^64-1. This is a fundamental Dart platform
limitation. Document this in the API doc for both methods.

**float precision loss:** `serializeFloat(double value)` truncates Dart's
64-bit double to C's 32-bit float. `deserializeFloat()` returns the 32-bit
result widened back to double. For example, `serializeFloat(10.1)` followed
by `deserializeFloat()` returns ~10.100000381, not 10.1. Tests should use
float-appropriate tolerance (~0.01 absolute for moderate values).

**`slices` getter:** Returns a lazy `Iterable` backed by `z_bytes_slice_iterator_t`.
Each element is a `Uint8List` representing one contiguous slice of the
potentially fragmented payload. For non-fragmented payloads, yields a single
element equivalent to `toBytes()`.

## CLI Example

### `package/example/z_bytes.dart`

Mirrors `extern/zenoh-c/examples/z_bytes.c`. No network — pure serialization
validation.

```
Usage: z_bytes [no options]
```

**Behavior:**
1. **Raw bytes** — `Uint8List → ZBytes → toBytes()`, assert equality
2. **String** — `String → ZBytes → toStr()`, assert equality
3. **Arithmetic** — `uint32 → serialize → deserialize`, assert equality
4. **Writer** — write chunks → finish → read back via slice iterator, assert
5. **Sequence** — serialize sequence of `(int32, string)` pairs → deserialize, assert
6. **Custom struct** — serialize `(float, uint64[][], string)` → deserialize, assert
7. **Slice iterator** — append 3 ZBytes via writer → iterate slices, print each
8. Print all results as PASS/FAIL

**zenoh-c example sections mapped:**

| z_bytes.c section | z_bytes.dart section |
|-------------------|---------------------|
| Raw buf + string wrapping | 1, 2 |
| `ze_serialize_uint32` / `ze_deserialize_uint32` | 3 |
| `z_bytes_writer_*` / `z_bytes_reader_*` | 4 |
| Sequence of `int32_t[]` | 5 |
| Sequence of `kv_pair_t` | 5 |
| Custom struct serialization | 6 |
| Slice iterator | 7 |

Note: The C example uses `z_bytes_reader_*` for section 4. Since we defer
the reader API, the Dart example reads back via `toBytes()` instead.

## Deferred

- **`z_bytes_reader_*` (seek/tell/read)** — Advanced random access into byte
  payloads. `ZBytes.toBytes()` handles full reads; the slice iterator handles
  fragmented access. Reader adds seek/tell which is uncommon for Dart use cases.
- **Typed list serialization helpers** — The C++ has template-based
  `vector<T>` / `map<K,V>` serialization. Dart can compose these from
  `serializeSequenceLength` + per-element calls. Sugar methods for
  `List<int>` / `Map<String, dynamic>` can be added later.
- **Android** — No platform-specific considerations. Serialization is
  pure computation, no POSIX dependencies.

## Overview

### Purpose
Implement a declared publisher -- a long-lived entity that efficiently publishes multiple messages to the same key expression with configurable QoS (encoding, congestion control, priority). This is the primary data-producing API for zenoh-dart, enabling applications to continuously publish data on a topic without the overhead of per-put key expression resolution.

### Use Cases
- Counter app publishing int64 values at high frequency on a fixed key
- IoT sensor publishing JSON data with application/json encoding
- Real-time data stream with priority and congestion control settings
- Monitoring subscriber presence via matching status listener

### Context
Phase 0-2 established the foundation: Session open/close, KeyExpr validation, ZBytes payload, one-shot put/delete, and callback-based subscriber via NativePort bridge. Phase 3 adds the first "declared entity" that persists across multiple operations, following the entity lifecycle pattern (sizeof -> declare -> loan -> operations -> drop/close) documented in the cross-cutting patterns doc.

## Implementation Details

### Architectural Approach

**Three-layer architecture (unchanged):**
1. C shim (`src/zenoh_dart.{h,c}`) -- 8 new `zd_` functions with flattened parameters
2. Generated FFI bindings -- auto-generated by ffigen
3. Idiomatic Dart API -- Publisher class, pure Dart types

**Key patterns:**
- Encoding is string-passthrough: Dart sends MIME type as `const char*`, C shim calls `z_encoding_from_str()`. Avoids 55+ C wrappers.
- Publisher keyExpr returns loaned keyexpr pointer, Dart extracts via existing helpers.
- Matching listener declared separately from publisher (two FFI calls, matching zenoh-c pattern).
- Matching status query uses out-parameter: `zd_publisher_get_matching_status(publisher, int* matching)`.

### Design Patterns
- **Entity lifecycle**: sizeof -> declare -> loan -> operations -> drop/close (Section 6 of cross-cutting patterns)
- **NativePort callback bridge**: Reused from Phase 2 subscriber for matching status listener
- **Flattened C shim parameters**: Sentinels (-1 for default enums, NULL for optional strings)

## Known Limitations / Trade-offs

### Limitations
- Per-put QoS override not supported (QoS is set at publisher declaration time, per zenoh-c design)
- `is_express`, `reliability`, `allowed_destination` options deferred to future phases
- `timestamp` and `source_info` on put/delete options deferred to Phase 9+

### Trade-offs Made
- **Encoding as String, not Encoding object on receive side:** Sample.encoding is `String?` not `Encoding?`. The receive side gets a MIME string from `z_encoding_to_string()`. Wrapping it back into an Encoding object adds overhead with no benefit -- callers compare strings anyway.
- **Matching listener declared separately:** Two FFI calls instead of one combined call. Matches zenoh-c/zenoh-cpp pattern and avoids overloading `zd_declare_publisher` with 7+ params.
- **Pure Dart Encoding class:** Avoids 55+ C wrapper functions for predefined encoding constants. Trade-off: C shim must parse the string with `z_encoding_from_str()` on each call.

## Implementation Notes

### Key Decisions
- **6-param zd_declare_publisher:** No dart_port for matching listener. Listener declared separately via `zd_publisher_declare_background_matching_listener`.
- **Out-parameter for matching status:** `zd_publisher_get_matching_status(publisher, int* matching)` returns error code, fills `*matching` with 0/1. Avoids conflating return value with status.
- **Loaned keyexpr for publisher.keyExpr:** Returns `const z_loaned_keyexpr_t*`, Dart extracts string via existing `zd_keyexpr_as_view_string` + `zd_view_string_data/len`. No buffer-copy.
- **ffigen.yaml opaque types:** Must add mappings for `z_owned_publisher_t`, `z_loaned_publisher_t`, `z_moved_publisher_t`, `z_owned_closure_matching_status_t`, `z_moved_closure_matching_status_t`, `z_matching_status_t`.

### Future Improvements
- Phase 9+: Add `is_express`, `timestamp`, `source_info` options
- Phase 9+: Extend `Session.put`/`Session.deleteResource` with encoding/QoS/attachment params
- Matching status could be extended with more detailed status info (subscriber count, locality)

### Potential Refactoring
- `_withKeyExpr` helper in Session could be reused for declarePublisher's key expression lifecycle (decide at implementation time)
- NativePort bridge pattern may be extractable into a shared helper after Phase 3 confirms it works for matching status (simpler Int64 post vs subscriber's array post)
